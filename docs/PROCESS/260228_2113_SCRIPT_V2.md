# DISPATCH — 45-Second Technical Demo Script v2

## Core demo principle

Everything is **scripted but live-rendered**.
You are running a **deterministic orchestrated simulation** that visibly exercises:

- Multilingual intake (Spanish, Mandarin)
- Shared evolving state with audit trail
- CCTV scene monitoring with vision AI
- Multi-agent coordination (7 agents, visible in terminal)
- One human approval moment
- **Vision detects what callers don't see** (fire — neither caller mentions it)
- **System warns all callers in their native language** (evacuation via ElevenLabs TTS)
- Evidence fusion across modalities (voice + voice + video)
- Clean final resolved state

---

## The narrative arc in one sentence

A crash comes in, an operator approves the response, a second caller reveals a child, the camera catches fire that nobody reported, and the system warns everyone in their own language.

## CCTV Video — 4 × 15-Second AI-Generated Clips

Generate each clip so the **last frame of clip N is the first frame of clip N+1**. Stitch into one 60-second `crash_video.mp4`. Fixed overhead CCTV angle, static camera, no movement. Intersection that reads as urban (Market St & 5th St feel). Night or dusk lighting reads better on dark UI.

### Clip 1 — "Impact" (0:00–0:15)

| | |
|---|---|
| **Start frame** | Normal intersection. A sedan is crossing. A delivery truck is approaching from the cross street. |
| **Action** | At ~7 seconds, the truck T-bones the sedan at the intersection. Impact. Debris scatters. Dust/glass cloud. The sedan spins and comes to rest against a light pole or building corner. |
| **End frame** | Wreckage settled. Truck stopped at angle. Sedan crumpled against pole. Dust cloud settling. No smoke yet. People on sidewalk frozen/looking. |
| **Mood** | Sudden violence → eerie stillness. |

### Clip 2 — "Aftermath + Smoke" (0:15–0:30)

| | |
|---|---|
| **Start frame** | Exact copy of Clip 1 end frame. Wreckage settled. |
| **Action** | Pedestrians cautiously approaching. A thin wisp of white/grey smoke begins rising from the sedan's engine area around ~20s. Smoke slowly intensifies through the clip. One bystander appears to be on a phone (this is our Caller 2). |
| **End frame** | Moderate smoke plume from engine. Clearly visible but no flames. Bystanders standing 15–20 feet back. |
| **Mood** | Tense concern. Something is wrong but not yet catastrophic. |

### Clip 3 — "Fire Ignition" (0:30–0:45)

| | |
|---|---|
| **Start frame** | Exact copy of Clip 2 end frame. Moderate smoke. |
| **Action** | Smoke darkens from grey to black around ~33s. At ~37s, first visible orange flame appears from under the hood. By ~42s, flames are clearly licking up from the engine compartment. Bystanders step further back. |
| **End frame** | Active engine fire. Orange/yellow flames clearly visible through smoke. Bystanders retreating. |
| **Mood** | Escalation. This just became much worse. |
| **KEY FRAME: t=38s** | Vision agent extracts this frame. Fire clearly starting. |

### Clip 4 — "Full Fire" (0:45–0:60)

| | |
|---|---|
| **Start frame** | Exact copy of Clip 3 end frame. Active fire. |
| **Action** | Fire intensifies. Flames spreading. Heavy black smoke. Bystanders well back. Scene clearly requires fire response. |
| **End frame** | Full emergency scene. Video can loop or hold on final frame. |

### Critical extraction frames for the orchestrator

| Frame | Video timestamp | What's visible | What agent detects |
|-------|----------------|----------------|-------------------|
| Frame 1 | t=25s | Moderate smoke, no fire | `smoke_visible: true, fire_visible: false, vehicle_damage: severe` |
| Frame 2 | t=38s | Active flames from engine | `engine_fire: true, confidence: 0.99` |

### The key narrative payoff

**Neither Caller 1 nor Caller 2 mentions fire or smoke.** Caller 1 reports the crash and a trapped husband. Caller 2 reports a child in the back seat. The CCTV is the only source that detects the fire. When the vision agent catches it, the system:

1. Corroborates the crash (callers said crash + camera confirms wreckage)
2. Detects a NEW hazard (fire) that no human reported
3. Fuses this with existing evidence (trapped adult + child + now fire = extreme danger)
4. Issues evacuation warnings to ALL callers in their native languages
5. Recommends fire response

This is the "the system sees what humans can't" moment. It's the hero beat of the entire demo.

## System architecture

### Backend components

- **DemoOrchestrator**: Drives timed events via `asyncio.sleep()` pauses
- **Supabase**: Canonical `incident_state`, `agent_logs`, `transcripts`, `dispatches` tables
- **Supabase Realtime**: Pushes all state changes to frontend automatically
- **Triage Agent** (`mistral-large`): Structured severity classification and action plans
- **Intake Agent** (`mistral-large`): Extracts structured facts from transcripts
- **Vision Agent** (`pixtral-12b`): Frame analysis + scene delta detection
- **Evidence Fusion Agent** (`mistral-large`): Cross-source corroboration
- **Case Match Agent** (`mistral-large`): Correlates multiple callers to same incident
- **Dispatch Agent** (`mistral-large`): Generates dispatch briefs
- **Voice/TTS Service** (`elevenlabs`): Transcription (Scribe) + multilingual voice responses

### Frontend panels

- **CCTV Panel** (bottom-left): Live video feed with bounding boxes and detection overlays
- **Case File Panel** (top-left): Evolving incident summary, severity, hazards, action plan
- **Caller Transcript Panel** (top-center): Tabbed per caller, shows inbound + outbound messages
- **Agent Terminal** (bottom-center): Real-time log of all agent activity with model names
- **Deployments Panel** (right): Response unit cards with status
- **Action Button** (bottom-right): Three-state approval button

## Build-light implementation notes

- Use **pre-seeded audio files** for caller clips
- Use **pre-seeded AI-generated video** (4 × 15s clips stitched)
- Use **hardcoded frame extraction timestamps** (t=25s for smoke, t=38s for fire)
- Let the orchestrator trigger events **deterministically** via `asyncio.sleep()` pauses
- Cache model outputs where possible to reduce latency variance
- If a Mistral call is slow (>4s), the demo still works — the UI just waits for the next Supabase event
- The only thing that MUST feel live is the **state updating in the UI**
- ElevenLabs TTS audio can be **pre-generated and cached** on first run, then played from cache on subsequent demo runs for consistency
